1. Kafka

Apache Kafka is a distributed event streaming platform. It lets applications publish, store, and consume streams of records or events.

Producer application -> Kafka -> Consumer application

Example event:

{
  "orderId": "123",
  "status": "CREATED",
  "createdAt": "2026-06-30T10:00:00Z"
}
Core idea: producers write data to Kafka, Kafka stores the data, and consumers read the data from Kafka.

2. Benefits Apart From Distribution

Decoupling services

Producer and consumer do not need to call each other directly. The producer publishes an event, and consumers process it independently.

Traffic buffering

If many events arrive suddenly, Kafka can hold them while consumers process at their own pace.

Replay

Kafka stores records for a configured retention period. A fixed or new consumer can replay older events from a chosen offset.

Multiple consumers

The same event can feed payment, inventory, email, analytics, and audit systems without the producer calling all of them.

Async workflows

Non-immediate work like sending emails, analytics, and audit logging can happen after the request returns.

Durability

If a consumer is down, records remain in Kafka until retention expires. The consumer can resume from its committed offset.

3. API vs Kafka

Yes, you can use APIs instead of Kafka in many cases. The choice depends on whether you need an immediate response or asynchronous event processing.

Use API For Use Kafka For
Immediate request-response communication. Events that say something already happened.
Get user details UserRegistered
Check balance PaymentCompleted
Cancel payment OrderCreated
Simple rule: APIs are good for commands and questions. Kafka is good for facts and events.

4. Broker vs Client Communication

Kafka broker and Kafka client communicate using Kafka's own binary protocol over TCP. It is not HTTP or REST by default.

Kafka Client <---- TCP connection using Kafka protocol ----> Kafka Broker

Kafka Broker

A broker is a Kafka server. It stores topic partitions and handles requests from producers, consumers, admin clients, and Kafka Streams applications.

Default broker port is usually 9092.

Kafka Client

A Kafka client is any application that talks to Kafka.

Producer Consumer Admin Client Kafka Streams app

Producer Communication

Producer connects to bootstrap broker
Producer fetches metadata
Producer finds the partition leader
Producer sends record to the leader broker
Broker responds with success or failure

Consumer Communication

Consumer -> Broker: give me records
Broker -> Consumer: here are records

Producers push records into Kafka. Consumers pull records from Kafka.

5. Bootstrap Broker

A bootstrap broker is not a special broker type. It is a normal Kafka broker whose address the client uses to discover the cluster.

bootstrap.servers=broker-1:9092,broker-2:9092,broker-3:9092

When the client starts, it connects to one of these brokers and asks for metadata:

  • All broker addresses
  • Topic names
  • Partition counts
  • Partition leaders
  • Cluster information
If one bootstrap broker fails, the client tries another configured bootstrap broker. If all configured bootstrap brokers are down or unreachable, the client cannot discover the cluster.

Bootstrap Broker vs Regular Broker

Bootstrap Broker Regular Broker
A normal broker used for initial discovery. Any Kafka server in the cluster.
Returns metadata to clients. Stores partitions, accepts writes, serves reads, replicates data, and may coordinate consumer groups.
Does not permanently forward all traffic. Clients talk directly to the brokers that lead the needed partitions.

6. Partition Leader

A Kafka topic is split into partitions. Each partition has one leader broker. Producers and consumers normally talk to the leader for reads and writes.

Topic: orders
Partitions: 3

orders-0 leader = broker-1
orders-1 leader = broker-2
orders-2 leader = broker-3

If a producer sends a record with key customer-123, Kafka hashes the key and chooses a partition. If the chosen partition is orders-1, and orders-1 is led by broker-2, the producer sends the record to broker-2.

Producer -> broker-2 -> orders-1

Followers copy data from the leader. If the leader fails, Kafka can elect an in-sync replica as the new leader.

Producer Acknowledgement

Setting Meaning
acks=0 Producer does not wait for confirmation.
acks=1 Leader broker confirms the write.
acks=all Leader and required in-sync replicas confirm the write.
Common production-safe choice: acks=all.

7. Consumer Polling

Kafka consumers are pull-based. The consumer asks the broker for records using a poll or fetch request.

while (true) {
  records = consumer.poll(Duration.ofMillis(1000));

  for (record in records) {
    process(record);
  }

  commit offset if needed
}

This is not usually CPU intensive because polling is not tight busy waiting. The poll call can wait for data and return earlier when records are available.

Important Settings

Setting Purpose
poll timeout How long the client waits for data in one poll call.
fetch.min.bytes Broker waits until enough data is available before responding, unless the wait timeout happens.
fetch.max.wait.ms Maximum time broker waits before responding even if fetch.min.bytes is not reached.
max.poll.records Maximum records returned in one poll.
max.poll.interval.ms Maximum allowed time between poll calls before Kafka considers the consumer unhealthy.

On high-traffic topics, poll returns quickly with records. On low-traffic topics, poll waits and may return empty or with a few records.

8. Topics Across Brokers

A broker does not usually own an entire topic. A topic is split into partitions, and those partitions are distributed across brokers.

Kafka cluster:
broker-1
broker-2
broker-3

Topic: orders
Partitions: 6
Replication factor: 3

Possible placement:

orders-0 leader broker-1, replicas broker-2 broker-3
orders-1 leader broker-2, replicas broker-1 broker-3
orders-2 leader broker-3, replicas broker-1 broker-2
orders-3 leader broker-1, replicas broker-2 broker-3
orders-4 leader broker-2, replicas broker-1 broker-3
orders-5 leader broker-3, replicas broker-1 broker-2

Another View

broker-1:
  leader for orders-0
  follower for orders-1
  follower for orders-2

broker-2:
  follower for orders-0
  leader for orders-1
  follower for orders-2

broker-3:
  follower for orders-0
  follower for orders-1
  leader for orders-2
More precise wording: a topic is across brokers because its partitions and replicas are distributed across brokers.

9. Security Modes

Kafka communication can be plain or secured depending on configuration.

Mode Meaning
PLAINTEXT No encryption or authentication at the protocol layer.
SSL / TLS Encrypted communication.
SASL_PLAINTEXT Authentication without transport encryption.
SASL_SSL Authentication plus encrypted communication.
Production systems commonly use SASL_SSL.

10. Short Summary

  • Kafka uses a TCP-based binary protocol between clients and brokers.
  • A bootstrap broker is a regular broker used for initial cluster discovery.
  • Clients should configure multiple bootstrap servers.
  • A topic is split into partitions.
  • Partitions are distributed across brokers.
  • Each partition has one leader broker.
  • Producers write to the partition leader.
  • Consumers usually read from the partition leader.
  • Consumers pull data using poll/fetch requests.
  • Polling is not normally CPU-heavy because the request can wait and return batches.
  • Kafka is best for async events, buffering, replay, fan-out, durability, and decoupling.
  • APIs are best when the caller needs an immediate answer.
